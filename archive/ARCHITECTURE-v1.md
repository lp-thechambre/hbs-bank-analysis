## 二、数据文件层（不可违反）

### 2.1 数据结构

所有数据以**文件**形式存储在磁盘，不塞入 spawn 的 prompt。这是兼容小上下文模型（如 Gemma 4 32k）的前提。

```
data/YYYY-MM-DD/               ← 每次跑生成一个时间戳目录
├── index.csv                  ← 42 行，每行极简核心指标
├── cluster_report.json        ← embedding 聚类结果
├── cards/                     ← 每家银行完整数据卡片
│   ├── 600036.md
│   ├── 601398.md
│   └── ...
├── quant_markers.json         ← 定量 spawn 输出
├── edge_markers.json          ← 边缘信号 spawn 输出
├── qual_markers_*.json        ← 定性 spawn 输出（每组一个）
└── final_output.json          ← 统合 spawn 最终输出
```

### 2.2 index.csv 格式（不可违反）

每行控制在 **~50 tokens**，确保任何模型都能将整个 index 常驻上下文。

```
code,name,type,pb,roe,npl,car,nim,mcap_rank
600036,招商银行,综合型,0.72,10.8,0.95,16.3,2.12,1
601398,工商银行,传统型,0.55,11.2,1.25,17.8,1.89,2
...
```

**自由度**（Claude Code 可决定）：
- 增加哪些核心字段（建议 5-8 个，不宜太多）
- 缺失值用 `NA` 还是 `null`

### 2.3 卡片格式（自由发挥）

每张卡片 ~1000-1500 tokens。格式由 Claude Code 自行决定，但需满足：
- 字段齐全（42 家字段数一致，缺失标 `null`）
- 包含该行的财务指标解释性上下文（方便 AI 理解数字含义）
- 可选：包含该行最近季度的趋势（如"NPL 连续两季上升"）

**建议方向**（Claude Code 可发挥想象力）：
- 卡片是否加入该行的行业排名（"PB 同行第 3 低"）
- 是否加入简单的同比/环比变化指示
- 是否包含银行类型分类的行业基准值

### 2.4 聚类报告格式（自由发挥）

Claude Code 决定聚类方法和输出格式。建议包含：
- 聚类数（建议 3-5 簇）
- 每簇成员列表（银行代码）
- 每簇的典型特征描述（"高 PB 低 NPL"）
- 离群点银行清单（如有）

## 三、四层 Spawn 设计（不可违反）

### 3.1 调度 Spawn

**角色**：管道编排器。不分析任何银行数据，只负责在正确时间启动正确的事情。

**不可违反**：
- 隔离运行在 spawn 中，main session 只做触发
- 层间数据通过磁盘文件传递，不通过 spawn 上下文
- 管道总超时 20 分钟，超时输出当前已完成的最优结果
- 每层完成后写 markers 文件到磁盘

**自由度**：
- 定量和边缘信号是否并行启动（推荐是）
- 异步通知 vs 同步等待的机制选择
- 重试策略（指数退避次数）
- 是否用 cron 做超时监控（或 spawn 自带的 wait 机制）

**调度 Spawn 的 Prompt 骨架（示意，由 Claude Code 填充完整）：**

```
你是一个银行筛选管道的调度员。你的职责是：

1. 执行数据工程脚本（`scripts/fetch_financials.py` + `scripts/pb_fetcher.py`）
   → 确认 data/YYYY-MM-DD/ 目录生成完毕
2. 启动 embedding 聚类（调用 oMLX /v1/embeddings 或本地脚本）
   → 输出 cluster_report.json
3. 暂停第一轮：
   a) spawn 定量分析 spawn（1个，入参：data/ 路径）
   b) spawn 边缘信号 spawn（1个，入参：data/ 路径）
   两者可并行。
4. 各层完成后，收集 quant_markers.json + edge_markers.json
5. 暂停第二轮：根据定量结果决定定性分析的银行名单
   → spawn 定性分析 spawn（按类型分组，3-4个）
6. 收集所有 qual_markers_*.json
7. 暂停第三轮：spawn 统合 spawn（入参：所有 markers 路径）
8. 收集 final_output.json
9. 通知 main session：管道完成
```

### 3.2 定量 Spawn（1个）

**角色**：只跟数字对话。看 index.csv + 按需 read 卡片，识别指标组合中的模式。

**不可违反**：
- 上下文常驻：`index.csv`（~2k）+ 聚类报告（~300 tokens）
- 按需使用 `read` 工具读取完整卡片（不一次性读完全部 42 家）
- 可用 `memory_search` 搜历史筛选案例和规则文档
- 不替换现有的 `compute_scores.py`（作为可选参考，而非必须执行）
- 输出格式：`quant_markers.json`，每行一个银行，包含：code, status (PASS|WATCH|REJECT), confidence (高|中|低), curiosity（字符串，最多 100 字）

**自由度**：
- 具体关注哪些指标组合
- 好奇心标记的详细程度和表达方式
- 可使用 `memory_search` 搜索哪些参考规则
- 是否需要 embedding 做相似度辅助

**必须避免**：
- ⛔ 不写死评分公式或固定权重
- ⛔ 不因为某字段缺失就抛异常或瞎编

### 3.3 边缘信号 Spawn（1个）

**角色**：找出异常。全局视野，专找偏离正常模式的银行。

**不可违反**：
- 上下文常驻：聚类报告（~300 tokens）+ index.csv（~2k）
- 与定量 spawn 并行运行
- 输出格式：`edge_markers.json`，每个标记包含：code, anomaly_type, severity, description

**自由度**：
- 采用什么异常检测方法（聚类偏离、统计异常、embedding 距离）
- 是否需要额外工具（如 sklearn）
- 是否做与历史筛选记录的纵向对比
- 判断异常严重程度的标准

**必须避免**：
- ⛔ 不因为数据缺失就标记为异常（正常的 NA 不触发）

### 3.4 定性 Spawn（3-4个，按银行类型分组）

**角色**：按同行分组，做横向比较。带着定量和边缘的好奇心标记去深挖。

**不可违反**：
- 按银行类型分组（传统商业银行 / 股份行 / 城商行 / 农商行或自定义）
- 每 spawn 处理 4-8 家（确保任何模型上下文装得下）
- 输入：本组银行的完整卡片 + 定量/边缘的好奇心标记
- 输出格式：`qual_markers_{group}.json`，每家包含：code, assessment (PASS|WATCH|REJECT), note, 回应了哪些上游好奇心

**自由度**：
- 分类方式（按资产规模、按业务模式、按区域等）
- 组内横向对比的具体方法
- 是否利用 `memory_search` 搜索新闻/管理层动态
- 每组 spawn 的 prompt 是否根据组类型定制（如"这组是城商行，特别关注 NIM 和区域经济"）

### 3.5 统合 Spawn（1个）

**角色**：裁判+补充侦查。不重新分析数据，只做交叉判定和三面不一致的冲突解析。

**不可违反**：
- 输入：所有 markers JSON（定量 + 边缘 + 定性各组）
- 工作流：
  1. 读全部 markers → 自动分三组（高信心通过 / 一致淘汰 / 三面冲突）
  2. 高信心通过 → 自动入围（不细看）
  3. 一致淘汰 → 自动排除（不细看）
  4. **只审三面冲突的银行**（通常 8-12 家）
  5. 在冲突组中按**冲突类型批量处理**，不是逐家串行
  6. 统合 spawn 可做**补充侦查**——对有模式裂缝的银行做进一步检索
- 输出：`final_output.json`，包含 10-15 家 final candidates，每家附来源追踪（哪个维度的什么标记导致这个结论）
- 统合 spawn 的上下文限制：读 markers 全文（~5k）+ 最多 read 5 张额外卡片

**自由度**：
- 冲突类型如何定义和分类
- 补充侦查的方式和范围
- 如果入围超过 15 家，如何做最终排序
- 输出格式和详细程度

**必须避免**：
- ⛔ 不推翻前三层的标记（只能补充，不能覆盖）
- ⛔ 不重新分析已被排除的银行
- ⛔ 不无限制 read 全部卡片
- ⛔ 不引入自己的评分体系

## 四、Embedding 使用场景（自由发挥）

现有的 Qwen3-Embedding-0.6B（oMLX :8000）接入方案由 Claude Code 自行设计，但需在以下场景中考虑应用：

| 场景 | 推荐度 | 说明 |
|------|--------|------|
| 银行卡片向量化 + 聚类 | **推荐** | 生成 cluster_report.json |
| 定性 spawn 检索历史案例 | **推荐** | `memory_search` 搜索上次筛选的类似银行 |
| 统合 spawn 的补充侦查 | **可选** | 搜索"NPL上升但ROE未降"等模式 |
| 银行相似度匹配 | **可选** | 让用户自然语言查询类似银行 |
| 规则文档 RAG | 低优先 | 因为规则已经精简到 spawn prompt 里了 |

**Claude Code 可自我决定**：
- 是否在数据工程阶段批量调用 embedding API
- 是否将聚类结果矩阵写入 cluster_report.json
- 各 spawn 是否以及在何时使用 `memory_search`
- 是否需要降维（如 PCA）可视化聚类结果

## 五、技术要点（不可违反）

### 5.1 文件读取规范

- 所有 spawn 的 `allowed-tools` 中必须包含 `read`
- 所有 spawn 的 `allowed-tools` 中不应包含 `exec`（只有调度 spawn 和统合 spawn 需要，视设计而定）
- 卡片路径通过入参传入，不硬编码

### 5.2 错误处理

- 任何脚本不因单个银行字段缺失而崩溃（缺失写 `null`，继续处理下一家）
- 任何 spawn 不因单家数据异常而中止整批分析
- 调度 spawn 必须捕获各层 spawn 的超时或失败

### 5.3 隔离要求

- 调度 spawn 运行期间 main session 恢复待命
- 数据仅存于 `data/YYYY-MM-DD/` 目录，不污染其他路径
- 每一层 markers 独立写入，避免全部丢失

1. **index.csv 的字段选择**——核心指标是你定
2. **卡片格式**——我给了范围，格式细节你设计

7. **memory_search 的使用时机和查询策略**——自由发挥
8. **数据工程的脚本改动量**——可以重用现有的 `fetch_financials.py`，可以部分改写，也可以新增辅助脚本

**红线再强调一次：**
- 不改三维度分层架构
- 不改四层 spawn 拓扑（调度→[定量+边缘]→定性→统合）
